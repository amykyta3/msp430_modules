Detailed Description

API for MSP430's USB peripheral. Supports HID-Datapipe, Traditional HID and CDC.

Author
Links to original USB API by Texas Instruments
Modified by Alex Mykyta to work with personal library

MSP430 Processor Families Supported:

  • MSP430x5xx
  • MSP430x6xx

Compilers Supported:

  • TI CCS v4
  • MSPGCC

Implementation Suggestion:
The following code should be placed in the initialization routine event to properly set up the USB API:

USB_init();
// Enable various USB event handling routines
USB_setEnabledEvents(USBEV_ALLEVENTS_EN);
// See if we're already attached physically to USB, and if so, connect to it
if (USB_connectionInfo() & USB_VBUS_PRESENT){
if (USB_enable() == 0){
USB_reset();
USB_connect();
}
}

The following code can be used in the onIdle() routine in order to poll the USB state:

switch(USB_connectionState()){
case USBST_DISCONNECTED:
// This state indicates that VBUS is not present, meaning there is no USB connection.
break;
case USBST_CONNECTED_NO_ENUM:
// This state indicates that VBUS is present but USB_connect() has not yet been called
break;
case USBST_ENUM_ACTIVE:
// This state indicates that the device has successfully enumerated on a USB host and is
// connected and active.
// The MSP430 should not be placed into a mode below LPM0 (no LPM3/4/5).
break;
case USBST_ENUM_SUSPENDED:
// This state indicates the device is enumerated on a USB host, but the host has suspended it.
// During suspend, the power mode may enter LPM3 or LPM4.
break;
case USBST_ENUM_IN_PROGRESS:
// This state indicates the device is in the process of being enumerated on a USB host.
// The MSP430 should not be placed into a mode below LPM0 (no LPM3/4/5).
break;
case USBST_NOENUM_SUSPENDED:
// This state indicates that VBUS is present, the device attempted to enumerate by pulling
// PUR high, but then the host suspended the device.
break;
case USBST_ERROR:
break;
}

RES_t USB_cdcIrecv ( void *  dst,
uint16_t  size,
uint8_t  intfNum 
)

Initiate a receive operation over the USB CDC. (Immediate)

Function initiates a receive operation and returns immediately. The USB API begins fetching data from the dst buffer. Data in the dst buffer is not valid until the receive operation completes. This can be checked by polling the status using USB_cdcStatus()

Parameters
[in]dstPointer to the destination buffer
[in]sizeNumber of bytes to receive
[in]intfNumInterface number
Return values
RES_OKReceive operation started successfully
RES_BUSYAnother receive operation is already in progress.
RES_FAILReceive operation failed.

References RES_BUSY, RES_FAIL, and RES_OK.

◆ USB_cdcStatus()

uint8_t USB_cdcStatus ( uint8_t  intfNum)

Checks the status of a USB interface.

Parameters
[in]intfNumInterface number
Returns
Byte containing bit-flags which indicate the status of the USB API
Return values
USB_ISENDACTIVEBit indicates that a send operation is open on this interface
USB_IRECVACTIVEBit indicates that a receive operation is open on this interface
USB_DATAWAITINGBit indicates that data has been received from the host for this interface, waiting in the USB receive buffers, lacking an open receive operation to accept it.
USB_BUSNOTAVAILBit indicates that the bus is either suspended or disconnected. Any operations that had previously been underway are now aborted.

RES_t USB_hidSendReport ( void *  reportData,
uint8_t  intfNum 
)

Sends a pre-built report to the host.

Parameters
[in]reportDataPointer to HID report data
[in]intfNumInterface number
Return values
RES_OKData has been written to the USB transmit buffers, and will be transferred to the host in the next polling frame
RES_BUSYUSB buffer for the interface has data in it, suggesting that the host has not yet fetched the previously-loaded report
RES_FAILThe bus has either been disconnected or the device is suspended, allowing no reports to be sent

reportData must be organized to reflect the format defined by the report descriptor in descriptors.c.

References RES_BUSY, RES_FAIL, and RES_OK.
